     Job Dashboard UI Redesign ‚ÄĒ Drawer + Row Quick-Change

     Context

     Today the list page sends users to a full-page route (/jobs/:id) to do
     anything beyond reading a job's name. The status-update flow is two-step
     (select ‚Üí Apply). Three problems:

     1. Only the job name is clickable, despite the whole row reading like a
     target.
     2. Navigating away from the list loses scroll/cursor position and forces a
     refetch on return ‚ÄĒ bad for browsing many jobs.
     3. The two-step status update is more friction than a one-click flow needs
     to be.

     Goal: keep the distinct shareable URL per job (preserves the take-home
     stretch goal) while turning the detail view into a right-side drawer
     overlaid on the list. Add a per-row status quick-change popover so the most
      common action (status update) doesn't require opening the drawer at all.
     Make all destructive actions go through a true confirm modal.

     Approach

     Routing (React Router 7 nested + Outlet)

     router.tsx: convert /jobs into a parent route with a child :id route.

     /jobs           ‚Üí JobsListPage (renders <Outlet />)
     /jobs/:id       ‚Üí JobDetailDrawer (rendered into the Outlet)

     - List stays mounted as parent ‚Üí no list refetch / scroll loss when opening
      detail.
     - Drawer open state = "child route is matched". Closing the drawer =
     navigate('/jobs').
     - Direct deep links to /jobs/:id still work ‚ÄĒ list mounts in the
     background, drawer opens on top.
     - Delete JobDetailPage.tsx. The drawer replaces it entirely.

     Drawer primitive (new, no shadcn)

     features/jobs/JobDetailDrawer.tsx ‚ÄĒ wraps existing JobDetail content.
     Custom (no new deps), built on a portal:

     - role="dialog" + aria-modal="true" + aria-labelledby (job name).
     - Right-anchored panel: w-full md:w-[480px], full-height. Backdrop dims the
      list. Slide-in transition (translate-x), single high-impact motion moment.
     - Mobile: full-width sheet (no backdrop margin). Desktop: 480px panel + dim
      backdrop.
     - Focus trap while open; Escape closes; backdrop click closes; close button
      in header.
     - Returns focus to the triggering row on close (track via data-job-id on
     row ‚Üí focus on close).
     - Body scroll lock while open.

     Row clickability

     JobList.tsx:

     - Desktop table: wrap each <tr> with click handler that does
     navigate('/jobs/:id'). Add role="link" + tabIndex={0} + Enter/Space
     keyboard handling so it stays a single clickable region. Visual:
     hover:bg-rescale-* and focus-visible ring.
     - Mobile cards: same ‚ÄĒ make <article> the click target with the same a11y
     wrapper.
     - Hit-test exception: status pill button and kebab button stop propagation
     so they don't also open the drawer.

     Status quick-change popover (new)

     features/jobs/StatusQuickChange.tsx ‚ÄĒ replaces the static <StatusPill> on
     rows with a button-styled pill:

     - Trigger: <button> styled like the current pill, with a ‚Ėĺ chevron.
     - Popover: role="menu" with one role="menuitemradio" per status. Arrow keys
      navigate, Enter/Space select, Escape closes.
     - Click a status ‚Üí fires useUpdateStatus optimistically (update keys.list
     cache + keys.detail(id) if cached) and closes immediately.
     - On error: roll back optimistic write + show inline ErrorBanner
     (role="alert") anchored to the row.
     - No toast/undo (avoids new infra; keeps to existing patterns).
     - The current StatusUpdateControl (radiogroup + Apply) is removed from the
     drawer ‚ÄĒ drawer's status section becomes a read-only StatusPill + a single
     "Change status" button that opens the same popover anchored from the drawer
      header. Single source of truth for the status-change mechanism.

     Delete (modal confirm, two entry points)

     features/jobs/ConfirmDeleteDialog.tsx ‚ÄĒ true modal (replaces today's
     disclosure):

     - role="dialog" + aria-modal="true", focus trapped, Escape closes, backdrop
      dims, Cancel focused by default (not Delete ‚ÄĒ destructive default is
     unsafe).
     - Title: "Delete job?". Body: name + "This will permanently remove the job
     and its full status history."
     - Buttons: Cancel (default) and Delete (red). Delete fires useDeleteJob; on
      success closes modal + navigates to /jobs (closing the drawer too).

     Two entry points, both open this modal:

     1. Drawer footer: "Delete job" button.
     2. Row trailing kebab menu (features/jobs/JobRowMenu.tsx): ‚čĮ button ‚Üí menu
     with single "Delete‚Ä¶" item. Stops row-click propagation. Same modal.

     The modal lives at the page level (rendered once in JobsListPage), opened
     via local state; the drawer and row-menu both call the same opener with the
      target job. Keeps focus return logic single-sourced.

     Sort (new ‚ÄĒ addresses stretch goal)

     Today GET /api/jobs/ enforces static -created_at, -id ordering because
     Ninja's CursorPagination doesn't accept user-controlled sort (verified in
     CLAUDE.md). To add sort without abandoning cursor pagination:

     Backend ‚ÄĒ backend/jobs/api.py + a small custom paginator:

     - Accept sort query param: created_at | -created_at | updated_at | 
     -updated_at | name | -name. Default -created_at. Validate via Literal enum
     (no from __future__ import annotations in this module ‚ÄĒ see CLAUDE.md
     gotcha).
     - Custom SortableCursorPagination(CursorPagination) subclass: reads sort
     from request, builds ordering as (sort_field, "id" or "-id") (matching
     direction for stable tie-break), passes through to CursorPagination. Cursor
      stays opaque to clients; embeds the sort field's value + id.
     - Composite indexes added in a Django migration: (updated_at DESC, id 
     DESC), (name ASC, id ASC). created_at already covered by existing index.
     Names ‚Č§30 chars: idx_job_updated_id_desc, idx_job_name_id_asc.
     - BE unit test: each sort option returns deterministic order across page
     boundaries (seed 30 jobs with overlapping timestamps + names; assert no
     dupes/skips when paging).

     Frontend ‚ÄĒ sort lives in URL state on /jobs?sort=... so it's shareable +
     back-button friendly:

     - JobList.tsx desktop table: column headers (Name, Created, Updated) become
      <button>s inside <th> with aria-sort="ascending|descending|none". Click
     toggles direction; clicking a different column resets to that column's
     sensible default direction (name=asc, dates=desc).
     - Mobile (no table): a <select> "Sort by" + a direction toggle button above
      the card list. Same URL state.
     - Sort changes invalidate keys.list (the existing factory already keys on
     filter/cursor; extend to include sort so caches don't collide). Reset
     cursor to first page on sort change.
     - E2E: frontend/e2e/sort.spec.ts ‚ÄĒ sort by name asc, assert order; toggle
     to desc; switch to updated_at; deep-link ?sort=name opens correctly.

     Mutation invalidation (preserves CLAUDE.md rules)

     - useUpdateStatus (optimistic now): onMutate does
     setQueryData(keys.list({status, cursor}), ...) +
     setQueryData(keys.detail(id), ...) if present. onError rolls back.
     onSettled invalidates ['jobs','list'] + ['jobs','history',id] (NOT umbrella
      ‚ÄĒ would clobber cached detail).
     - useDeleteJob: invalidate umbrella ['jobs'], navigate /jobs.
     - useCreateJob: unchanged.
